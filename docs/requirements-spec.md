## 1. 项目背景

### 1.1 行业趋势

招投标行业正经历智能化变革：

- **政策驱动**：国务院明确"加快 AI 在公共资源招投标活动中的应用"；国家发改委等部门 2026 年印发《关于加快招标投标领域人工智能推广应用的实施意见》，要求 2026 年底前在部分省市实现"招标文件检测、智能辅助评标、围串标识别"等重点场景全覆盖。
- **AI 评标落地**：行业从"经验驱动 / 关系驱动 / 模板驱动"转向"规则驱动 / 数据驱动 / 合规驱动"。
- **传统编标痛点**：百万级招标项目需 3-5 人苦战数周，细节疏忽即废标。
- **市场窗口**：亟需一款真正理解招投标规则、能辅助投标人高效合规完成标书编制的 AI 工具。

### 1.2 产品定位

**面向投标方**、**覆盖投标文件编制全流程**的 AI 智能辅助工具。

- **核心闭环**：`读懂招标文件 → 智能生成标书 → 全维度合规审查`
- **核心价值**：将专家从 80% 的低效重复劳动中解放出来，聚焦于技术方案优化与竞争策略制定。
- **战略 15 字诀**：`读得准 / 写得实 / 审得全 / 学得进 / 看得美`

### 1.3 用户画像

| 角色 | 工作内容 | 痛点 | 期望收益 |
|---|---|---|---|
| 资深投标专家 | 统筹标书编制、把控质量、对接客户 | 时间被重复劳动吞噬；废标风险压力大 | 把精力集中在 20% 高价值创造工作 |
| 编标专员 | 撰写技术方案、整理素材、排版 | 重复劳动占比高；易出低级错误 | 提速、降错率；获得辅助建议 |
| 商务专员 | 处理资质、报价、合同条款 | 信息分散；前后不一致 | 知识库自动调用 + 一致性校验 |
| 项目经理 | 协调分工、跟进进度、把控风险 | 多人协作冲突；版本混乱 | 任务分配、版本管理、审计追踪 |
| 评审/复核人 | 检查合规性、查重 | 人工核校耗时且易疲劳 | 全维度审查 + 分级风险提示 |

## 2. 痛点分析

需求书原文归纳的五类痛点，构成后续功能模块的需求来源：

### 2.1 招标文件解读之痛

- 招标文件动辄数百页，含大量隐性要求。
- 专家手动精读一份招标文件往往需数天，仍易遗漏藏在备注、脚注、附件中的关键信息。
- ★号条款、暗标格式要求、评分权重分布等关键信息，人工提取效率低且易出错。

**对应需求**：3.1 招标文件智能解析模块。

### 2.2 标书编制之痛

| # | 痛点 | 描述 |
|---|---|---|
| 1 | 重复劳动占比过高 | 反复复制粘贴通用模板、改年份改项目名、熬夜排版调格式；占编标 80% 时间 |
| 2 | 内容针对性不足 | 标书内容空泛；招标方想要的内容看不到；所需资料不齐 |
| 3 | 结构随意导致丢分 | 章节错位、要点分散、内容穿插；AI 评标定点抓取关键词，放错位置直接判未响应 |
| 4 | 数据不一致隐患 | 前后参数不一致、业绩时间冲突、服务周期矛盾、资质描述不统一 |
| 5 | 图表制作耗时且位置难把握 | 优质图表耗时是同等文字数倍；图表位置不当（太早/太晚）影响评审体验；导致"有文无图" |

**对应需求**：3.2 智能标书生成 + 3.7 智能图表生成与插叙。

### 2.3 合规审查之痛

- **废标风险无处不在**：投标函人员与表格不一致、人员数量不满足、格式不规范、签字盖章不全。
- **低级错误防不胜防**：复制粘贴出错、采购人名称写错、公用文件忘替换、错盖公章。
- **人工审查效率低**：几百页标书逐字逐句核对，不仅耗时，且极易疲劳出错。

**对应需求**：3.4 全维度合规智能审查模块。

### 2.4 素材管理之痛

- 历史标书、资质证书、业绩案例、人员信息分散在不同部门和系统中。
- 编制时需"翻箱倒柜"查找；缺乏统一知识库导致重复工作、信息不一致。

**对应需求**：3.3 企业知识库管理模块。

### 2.5 AI 评标适配之痛

- 当前主流 AI 评标系统：不懂语感、不包容模糊、不接受"大概 / 差不多"。
- 人工写标是写给人看的，AI 评标是读机器数据的。
- 未来标书必须满足**三个机器适配标准**：
  - **结构标准化**
  - **内容定制化**
  - **数据结构化**
- 纯人工制作几乎无法同时满足这三点。

**对应需求**：3.5 评分响应与优化 + 3.7 智能图表"机器可读性"适配。

## 3. 功能性需求

### 3.1 招标文件智能解析模块

#### 3.1.1 需求描述

上传 PDF 或 Word 格式招标文件后，系统自动完成全文解析，提取核心信息并结构化输出。

#### 3.1.2 功能要求

| # | 要求 | 说明 |
|---|---|---|
| FR-3.1-A | 多格式支持 | 接受 PDF（文本 + 扫描）、Word（.docx） |
| FR-3.1-B | 超长文档 | 支持百万字级别单文件解析 |
| FR-3.1-C | 自动信息提取 | 项目基本信息（名称、编号、招标人、预算等）、资质门槛（注册资本 / 资质等级 / 人员要求等）、技术参数与规格要求、评分细则与权重分布、★号条款与废标红线、投标文件格式与递交要求 |
| FR-3.1-D | 隐性要求高召回 | 对备注、脚注、附件中的隐性要求有较高召回率 |
| FR-3.1-E | 结构化目录 | 输出可检索目录；高亮标注核心得分点与废标风险点 |
| FR-3.1-F | 暗标识别 | 自动识别暗标要求（如封面不得出现公司名称、不得出现可识别投标人身份信息），并给出醒目提示 |

#### 3.1.3 性能指标

| 指标 | 目标值 |
|---|---|
| 解析耗时 | 百万字 ≤ 60 秒 |
| 解析准确率 | ≥ 95% |

#### 3.1.4 验收标准

- ✅ 上传一份 100 万字 PDF 招标文件，60 秒内返回完整结构化目录。
- ✅ ★号条款 100% 召回（漏召回即视为缺陷）。
- ✅ 自动识别的暗标要求必须在结果中**醒目高亮**（非平铺）。
- ✅ 拿到解析结果的专家**不需要再回头翻原文件**，即可掌握全部关键信息。

### 3.2 智能标书生成模块

#### 3.2.1 需求描述

基于招标文件解析结果和企业知识库，自动生成符合要求的投标文件初稿。

#### 3.2.2 功能要求

| # | 要求 | 说明 |
|---|---|---|
| FR-3.2-A | 智能大纲生成 | 运用自然语言处理与模板框架，依项目需求一键生成逻辑合理的投标大纲；大纲结构严格对标评分细则 |
| FR-3.2-B | 分模块生成 | 区分商务标 / 技术标 / 报价方案，对照招标要求逐条响应 |
| FR-3.2-C | 智能内容填充 | 学习分析历史数据；关联企业专属资质库；自动填充营业执照、专利证书、过往业绩等核心信息 |
| FR-3.2-D | 多行业适配 | 预置工程建设、政府采购、IT 信息化、医疗设备等高频行业的素材库与写作逻辑；从大纲结构、技术要点、法规引用上体现差异 |
| FR-3.2-E | 生成效率 | 百页标书生成时间不超过 10 分钟 |
| FR-3.2-F | 篇幅智能分配 | 根据评分权重自动建议各章节的篇幅投入：评分权重高的章节重点展开，低权重简要处理 |
| FR-3.2-G | 行业专业性 | 核心方案需体现行业专业性，非通用套话 |
| FR-3.2-H | 图文并茂 | 不应是"通篇文字墙"，而是图文并茂、逻辑清晰、评审友好的专业文档 |

#### 3.2.3 性能指标

| 指标 | 目标值 |
|---|---|
| 百页标书生成时间 | ≤ 10 分钟 |

#### 3.2.4 验收标准

- ✅ 生成的标书**不是填空式模板**，而是真正读懂招标文件、结合企业真实信息的有针对性初稿。
- ✅ 核心方案具备行业专业性（评审专家一眼能识别行业）。
- ✅ 大纲与评分细则逐条对应（响应清单可追溯）。

### 3.3 企业知识库管理模块

#### 3.3.1 需求描述

建立企业专属的知识库，作为 AI 生成内容的数据来源。

#### 3.3.2 功能要求

| # | 要求 | 说明 |
|---|---|---|
| FR-3.3-A | 素材类型 | 上传过往中标标书、公司资质证书、业绩案例、人员信息、设备清单、技术方案模板 |
| FR-3.3-B | 历史标书导入 | 支持历史标书导入与解析；提取可复用结构化素材 |
| FR-3.3-C | 持续更新 | 知识库持续更新与维护；**资质证书到期自动提醒** |
| FR-3.3-D | 优先调用自有素材 | AI 生成内容优先调用企业自有素材，减少通用套话占比 |
| FR-3.3-E | 权限分级 | 不同角色可见不同级别的知识库内容 |
| FR-3.3-F | 图表素材库（新增） | 自动提取历史标书中的图表（技术架构图、流程图、组织架构图、甘特图等）；标注图表类型、适用场景、关键词、所属行业；构建企业专属图表素材库；新标书生成时优先匹配复用 |
| FR-3.3-G | 自我进化 | 中标案例的数据反哺模型，实现自我优化；知识库"越用越聪明" |

#### 3.3.3 验收标准

- ✅ 资质证书在到期前 N 天自动提醒（N 由系统配置，建议 90 天）。
- ✅ 每次投标的经验都能沉淀，而非每次从头再来。

### 3.4 全维度合规智能审查模块

#### 3.4.1 需求描述

对标书全文进行自动化合规校验，识别各类废标隐患和风险点。

#### 3.4.2 功能要求

| # | 要求 | 说明 |
|---|---|---|
| FR-3.4-A | 法规案例库 | 内置国家招投标法规库与废标案例库 |
| FR-3.4-B-1 | 基础信息审查 | 投标函金额与报价表一致性；资质证书有效期；人员名字一致性 |
| FR-3.4-B-2 | 格式规范审查 | 字体、字号、行距、页边距；签字盖章完整性 |
| FR-3.4-B-3 | 实质性响应审查 | 是否响应 ★号条款、星号条款 |
| FR-3.4-B-4 | 逻辑一致性审查 | 前后参数一致性；业绩时间逻辑；服务周期匹配 |
| FR-3.4-B-5 | 查重与原创度审查 | 全文原创度检测；内容去重改写 |
| FR-3.4-B-6 | 暗标合规审查 | 自动检测是否暴露企业敏感信息；自动脱敏处理 |
| FR-3.4-B-7 | 图表数据一致性审查 | 图表中数据与正文描述是否一致（如正文工期 6 个月、甘特图是否也为 6 个月；正文描述系统包含 5 个模块、架构图是否也展示了 5 个模块） |
| FR-3.4-B-8 | 图表规范审查 | 图编号、图标题、图注是否完整规范；图表在文档中的位置是否合理（是否紧跟在对应描述之后而非随意堆砌） |
| FR-3.4-C | 分级风险输出 | 标记风险点并给出修改建议；按严重程度分级：**致命风险（必改）/ 重要风险（建议改）/ 轻微瑕疵（可选改）** |
| FR-3.4-D | 修改建议可操作 | 审查报告清晰、可操作；编标人员可快速定位与修正 |

#### 3.4.3 性能指标

| 指标 | 目标值 |
|---|---|
| 百页标书审查时间 | ≤ 10 分钟 |
| 废标风险检出率 | ≥ 90% |

#### 3.4.4 验收标准

- ✅ 审查模块是专家最信赖的"安全网"。
- ✅ 风险报告按"致命 / 重要 / 轻微"分级，每条都附可操作修改建议（如"未提供近 3 年审计报告，需补充至商务标第 5 章"）。

### 3.5 评分响应与优化模块

#### 3.5.1 需求描述

帮助投标文件精准命中评分点，提高技术标得分。

#### 3.5.2 功能要求

| # | 要求 | 说明 |
|---|---|---|
| FR-3.5-A | 评分权重高亮 | 解析招标文件时自动标注评分权重；高亮关键得分项 |
| FR-3.5-B | 响应清单 | 生成"响应清单"，对照初步评审（形式 / 资格 / 响应性审查）与详细评审（商务 / 技术评分）的不同要求逐条核对 |
| FR-3.5-C | 技术方案优化 | 辅助优化技术方案内容；匹配评委关注的要点（如工期合理性、安全预案完整性等） |
| FR-3.5-D | 机器可读性评估 | 对标书内容进行"机器可读性"评估；提示哪些表述过于模糊、缺少量化数据 |
| FR-3.5-E | 预评分能力 | 对标书内容做预评分；专家在投标前即可知道哪里还能优化、哪里已经达标 |

#### 3.5.3 验收标准

- ✅ 系统能告诉专家"这个项目技术方案占 40 分，你应该把主要精力放在第 3 章和第 5 章，因为这两章合计占 25 分"。

### 3.7 智能图表生成与插叙模块（★ 核心差异化）

#### 3.7.1 需求描述

在标书生成过程中，自动识别各章节的图表需求，生成匹配的专业图表，并智能插入到文档的恰当位置，实现真正的"图文并茂"。

#### 3.7.2 图表类型覆盖（≥ 10 种）

| # | 类型 | 典型场景 |
|---|---|---|
| 1 | 技术架构图 | 系统分层、模块关系、数据流向 |
| 2 | 业务流程图 | 业务流程走向、审批路径 |
| 3 | 甘特图 / 横道图 | 项目实施计划、工期安排、里程碑 |
| 4 | 组织架构图 | 项目团队结构、人员配置 |
| 5 | 时序图 | 系统交互顺序、接口调用关系 |
| 6 | 思维导图 | 技术方案框架、需求分解 |
| 7 | 饼图 / 柱状图 / 折线图 | 数据分析、资源配置、预算分布 |
| 8 | 技术参数对比表 | 产品参数逐项响应 |
| 9 | 评分响应表 | 评分细则逐条对标 |
| 10 | 概念示意图（文生图） | 技术原理示意、应用场景展示 |

#### 3.7.3 自动需求识别

| # | 要求 | 说明 |
|---|---|---|
| FR-3.7-3-A | 规则引擎匹配 | 建立"章节主题 → 图表类型"映射规则库。如"系统架构"章节 → 架构图；"项目实施计划" → 甘特图；"技术方案概述" → 流程图；"组织保障" → 组织架构图；"技术参数响应" → 技术参数对比表 |
| FR-3.7-3-B | LLM 语义判断 | 对无法通过规则匹配的章节，由 LLM 根据章节标题与正文内容动态判断最合适的图表类型；输出结构化需求描述（图表类型、数据来源、展示重点等） |
| FR-3.7-3-C | 图表需求清单 | 生成正文的同时，输出完整"图表需求清单"，供编标人员审核与调整 |

#### 3.7.4 数据驱动生成（防幻觉）

| # | 要求 | 说明 |
|---|---|---|
| FR-3.7-4-A | 数据来源严格 | 仅从招标文件解析结果 + 企业知识库中提取数据；严禁凭空编造 |
| FR-3.7-4-B | 三层架构 | LLM 生成图表代码 → 渲染引擎渲染 → 嵌入文档 |
| FR-3.7-4-C | 图表描述语言 | Mermaid / ECharts / Vega-Lite 混合 |
| FR-3.7-4-D | 样式模板 | 预设商务风、科技风、简约风等多套样式模板；可自定义企业专属样式 |
| FR-3.7-4-E | 来源可溯源 | 每个图表的数据点必须能追溯到知识库中的原始素材 |

#### 3.7.5 智能位置匹配与插入（四步法）

| 步骤 | 要求 |
|---|---|
| ① 自然语言标记 | 在正文生成过程中插入结构化占位标记：`[图表：系统架构图，展示模块 A、B、C 之间的数据流向]` |
| ② 语义匹配 | 扫描全文图表标记；按**主题相似度 / 数据一致性 / 上下文关联度**三个维度与已生成的图表库匹配 |
| ③ 自动插入 | 将匹配成功的图表插入标记位置；自动添加图注（图编号、图标题） |
| ④ 版式调整 | 根据标书整体排版风格调整图表尺寸与图文间距 |

#### 3.7.6 文生图补充方案

| # | 要求 | 说明 |
|---|---|---|
| FR-3.7-6-A | Prompt 自动生成 | 根据章节关键词自动生成高质量文生图提示词（含风格、构图、色彩倾向） |
| FR-3.7-6-B | 模型调用 | 调用文生图模型（如 FLUX、Stable Diffusion 等） |
| FR-3.7-6-C | 适用范围 | 主要用于"技术原理示意""应用场景展示""概念模型演示"等非精确数据驱动的可视化 |

#### 3.7.7 图文联动与动态更新

| # | 要求 | 说明 |
|---|---|---|
| FR-3.7-7-A | 变更检测 | 当正文关键数据（工期、人数、预算、模块数量等）变更时，系统自动检测并提示"相关图表数据需同步更新" |
| FR-3.7-7-B | 一键刷新 | 支持一键刷新所有关联图表——重新提取最新数据 → 重新生成代码 → 重新渲染 → 替换旧图表 |
| FR-3.7-7-C | 版本追溯 | 记录图表每次更新历史；便于编标人员追溯与版本对比 |

#### 3.7.8 素材库复用（与 3.3 FR-3.3-F 联动）

- 自动入库：历史标书使用过的优质图表（架构图、流程图、组织架构图、甘特图等）自动提取入库；标注图表类型、适用场景、关键词、所属行业、评分反馈（该图表是否帮助中标等）。
- 智能匹配复用：新标书生成时优先从素材库匹配可复用模板，再做数据替换与样式调整。
- 用户上传：支持手动上传自定义图表模板（企业专属组织架构图样式、标准化技术架构底座图等）。

#### 3.7.9 性能指标

| 指标 | 目标值 |
|---|---|
| 单张标准图表生成 | ≤ 30 秒 |
| 百页标书全量图表总生成 | ≤ 5 分钟 |

#### 3.7.10 验收标准

- ✅ 专家读到文字描述时，图表就在旁边或下方，不需要前后翻找。
- ✅ 图表数据与正文完全一致（杜绝"正文说工期 6 个月，甘特图画了 8 个月"）。
- ✅ 图表视觉风格专业统一，符合行业惯例。

## 5. 技术难点与研发约束

> 本节作为**设计文档必须显式响应的难点清单**。每条均映射到 HLD / framework 的对应章节。

### 5.1 招标文件的异构性与解析准确性

| 项 | 要求 |
|---|---|
| 难点 | 招标格式五花八门（PDF 扫描件、Word、图片）；排版风格各异；同一含义条款表述完全不同 |
| 约束 | OCR + 语义分析多模态解析；建设招投标领域专用语义理解模型；持续迭代条款模式库；引入人工标注与反馈机制 |
| 映射 | HLD §3 端到端流程 / §5 解析模块设计 |

### 5.2 "幻觉"内容的防控

| 项 | 要求 |
|---|---|
| 难点 | 大模型可能凭空捏造不存在的资质、业绩、专利号、行业标准；招投标对真实性零容忍 |
| 约束 | 严格限制 AI 素材来源（仅知识库）；受控写作机制（可写、可提醒、可校对，但绝不胡编）；所有生成内容必须可溯源；事实核查机制 |
| 映射 | HLD §4 章节任务编排 / §5 受控写作闸门 |

### 5.3 标书的差异化与查重防控

| 项 | 要求 |
|---|---|
| 难点 | AI 生成千篇一律会导致查重率爆表；可能因"不同投标人投标文件异常一致"被认定串通投标 |
| 约束 | 高质量行业素材库（而非廉价通用模板）；内容变体生成算法；内置查重检测；支持企业自有历史标书做专属模型训练 |
| 映射 | HLD §4 章节任务变体生成 / §5 查重机制 |

### 5.4 评标逻辑的精准适配

| 项 | 要求 |
|---|---|
| 难点 | AI 评标只认"结构化数据"；不同地区、不同项目评标标准差异巨大 |
| 约束 | "评分规则引擎"将评分细则转化为可执行规则；按"结构标准化 / 内容定制化 / 数据结构化"三标准输出；分行业、分地区评标规则库；机器可读性评分机制 |
| 映射 | HLD §4 评分响应与优化 / §7 审查模块 |

### 5.5 长文本处理与生成质量

| 项 | 要求 |
|---|---|
| 难点 | 标书动辄几百页、上百万字；大模型长文本处理易出现上下文丢失、前后不一致、逻辑断裂 |
| 约束 | 长文本拆解 → 子问题规划 → 多 Agent 并行 → 精炼回填；多智能体协同架构；"认知—匹配—生成—校验—复用"工程化闭环；思维链技术 |
| 映射 | HLD §3 端到端流程 / §4 章节划分与调度 |

### 5.6 知识库的构建与持续进化

| 项 | 要求 |
|---|---|
| 难点 | 每个企业的资质、业绩、人员、产品都不同且动态变化 |
| 约束 | 灵活的知识库数据结构；自动更新机制（资质到期提醒、新业绩自动归档）；支持企业历史标书导入训练专属小模型；中标案例数据反哺机制 |
| 映射 | HLD §3 知识库服务设计 |

### 5.7 数据安全与合规

| 项 | 要求 |
|---|---|
| 难点 | 标书涉及核心商业信息、财务数据、技术秘密；工具使用本身也需符合招投标法律法规 |
| 约束 | 私有化部署；完善的数据加密、权限控制、操作审计体系；输出符合《招标投标法》及实施条例；持续跟踪政策动向 |
| 映射 | HLD §12 安全设计 / §13 部署方案 |

|---|---|
| 难点 | 见 §5.9 六大挑战 |
| 架构 | 见 §3.7.4 三层架构 + §3.7.5 四步法 |
| 映射 | HLD §5 图表设计与实现（重点章节） |

### 5.9 图表生成与插叙的六大核心技术挑战

| # | 挑战 | 约束 | 映射 |
|---|---|---|---|
| 5.9-① | 数据真实性保障 | 严格限制数据来源；加入事实核查环节；图表中每个数据点可溯源 | HLD §5 受控写作闸门 |
| 5.9-② | 图表与文本语义对齐 | 建立"双向语义索引"：文本侧（需求类型 + 关键词 + 数据指纹）+ 图表侧（图表类型 + 主题关键词 + 数据指纹）；通过向量相似度计算实现双向匹配；阈值需大量样本调优 | HLD §5 双向索引 |
| 5.9-③ | 复杂图表自动布局 | 引入力导向布局 / 层次布局 / 正交布局；针对 10-50 节点、20-100 连线规模调优 | HLD §5 布局算法 |
| 5.9-④ | 图表的行业专业性 | 分行业的图表规范知识库；初期聚焦 2-3 个高频行业深度优化，再扩展（工程建设：GB/T 50326-2017；IT 信息化：分层架构 / 微服务 / 中台；医疗设备：行业数据流向规范） | HLD §5 行业适配 |
| 5.9-⑤ | 生成效率与质量平衡 | 优先级调度（核心得分章节优先高质量生成）；异步生成机制（前端先展示文字，图表后台逐个渲染）；缓存复用（同一项目多次使用的同类型图表） | HLD §4 调度策略 |
| 5.9-⑥ | 机器可读性适配 | 图表采用"矢量图 + 结构化元数据"双层存储——矢量图供人阅读；图表对应的结构化数据（数据表格、模块列表、关系描述）以隐藏文本或 XML 元数据形式嵌入文档；图注说明包含足够关键词与量化数据 | HLD §5 机器可读性 + §6 Word 输出 |
