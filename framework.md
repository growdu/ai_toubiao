# 标书 AI Agent 系统设计（章节化版 v2）

> 本文档是通用标书 AI 系统的设计纲要。它不绑定任何具体实现项目，但每一处抽象都对应到工程现实中可落地的组件。

---

# 〇、设计哲学摘要

| 抽象 | 本质 |
|---|---|
| 标书 | 章节集合 + 图表集合 + 响应矩阵 + 证据链 |
| 拆分单元 | 标书章节（不是任务，不是 Agent 类型） |
| 图表 | 一等公民，跟章节正文一起设计、一起生成、一起审计 |
| 目标态 | 输入 RFP + 背景材料 → 自动生成完整标书（含人在回路点） |
| 并行粒度 | 章节级并行 + 章节内串行 |
| 一致性 | 章节内自审 + 跨章节终审 |

---

# 一、问题重定义

## 1. 标书的本质：不是"一份文档"

一份标书在工程上**不是**一份连续文档，而是**四组结构化产物的集合**：

```
标书（BidPackage）
 ├── 章节正文（ChapterContent[]）
 ├── 图表库（Illustration[]）
 ├── 响应矩阵（ResponseItem[]）   ← 招标要求逐条响应
 └── 证据链（Evidence[]）         ← 支撑性材料追溯
```

每一组产物都有独立的生成逻辑、独立的存储结构、独立的审核标准。把标书当"文档"处理是国内现有工具的通病，导致 AI 只能做润色，不能做生产。

## 2. 为什么按"章节"拆分，而不是按"任务"或"Agent 类型"

| 维度 | 章节拆分 | 任务拆分 | Agent 类型拆分 |
|---|---|---|---|
| 拆分依据 | 标书天然结构 | AI 处理步骤 | AI 模型能力 |
| 并行可行性 | 高（章节间弱依赖） | 低（任务链强依赖） | 低（同一章节要多种 Agent） |
| 一致性粒度 | 合适（章级独立审计） | 太细（段级碎片化） | 太粗（跨章节混合） |
| 用户可干预性 | 高（按章节重做） | 低（任务链耦合） | 低（Agent 失败难定位） |
| 图表归属 | 天然（图表属于章节） | 模糊（图表归属任务？） | 模糊（图表归哪个 Agent？） |

**结论**：章节是标书系统的"自然原子"。按章节拆分，图表就天然归属于章节，审计就天然有边界，并行就天然成立。

## 3. 输入和输出

**输入**：
- 标书要求：RFP 文件（PDF/Word）+ 关键参数（项目名、采购方、截止时间）
- 背景材料：公司资质、历史标书、技术文档、案例、人员

**输出**：
- 完整标书（Word/PDF）：含目录、章节、图表、附录
- 标书摘要页：一页纸总结
- 中间产物：章节大纲、章节素材库、一致性审计报告、响应矩阵、证据链

**端到端目标**：用户上传 RFP + 背景材料，点击"开始生成"，等待若干分钟（人在必要的回路点确认），得到一份可直接签章的标书。

## 4. 自动化等级（诚实面对边界）

| 等级 | 含义 | 例子 |
|---|---|---|
| L1 完全自动 | 用户提交输入，系统自动出结果 | RFP 解析、章节撰写、图表生成 |
| L2 人在回路 | 系统自动 + 人工确认关键节点 | 跨章节审计、废标项扫描 |
| L3 半自动 | 系统辅助 + 人工主导 | 最终 Word 样式微调 |

MVP 阶段以 L2 为主，少数环节逐步向 L1 扩展。这是合规要求决定的，不是技术限制。

# 二、核心系统架构

```text
                          ┌─────────────────┐
                          │   用户输入       │
                          │ ┌─────────────┐ │
                          │ │ RFP + 材料   │ │
                          │ └─────────────┘ │
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │ 输入校验 + 解析  │  ← RFP 结构化、背景索引化
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │  章节规划器      │  ← 一次性：输出大纲 + 规格清单
                          │ (Planner)        │
                          └────────┬────────┘
                                   ↓
        ┌──────────────────────────┴──────────────────────────┐
        │            章节级并行处理（每章节独立流水线）        │
        │ ┌────────┐  ┌────────┐  ┌────────┐         ┌──────┐│
        │ │  Ch 1  │  │  Ch 2  │  │  Ch 3  │   ...   │ Ch N ││
        │ │        │  │        │  │        │         │      ││
        │ │撰+图+审│  │撰+图+审│  │撰+图+审│         │      ││
        │ └────────┘  └────────┘  └────────┘         └──────┘│
        └──────────────────────────┬──────────────────────────┘
                                   ↓
                          ┌─────────────────┐
                          │ 跨章节一致性审计 │  ← 术语统一、响应矩阵交叉
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │ 废标项 + 查重    │  ← 合规风险、历史雷同
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │   人在回路       │  ← 审计问题处理 + 样式微调
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │   标书汇总器     │  ← 输出 Word/PDF
                          └────────┬────────┘
                                   ↓
                          ┌─────────────────┐
                          │   完整标书       │
                          └─────────────────┘
```

**关键设计**：
- 章节级并发 + 章节内串行（章节内多步可共享上下文）
- 每个章节是一个独立的"撰写 + 配图 + 自审"单元
- 跨章节一致性是独立阶段（不阻塞章节并行）
- 人在回路点是**架构内组件**，不是补丁

# 三、章节级核心组件设计

## 1. 章节规划器（Chapter Planner）

**输入**：
- RFP 解析后的结构化需求树
- 用户指定的章节粒度（粗 / 中 / 细）
- 历史标书模板（可选）

**输出**：
- 章节大纲（树形）
- 每章节的"写作规格说明书"：
  - 章节标题、层级、目标字数
  - 章节类型（technical / business / compliance / price / case / management）
  - 写作风格（narrative / list / comparison / mixed）
  - 必含要素清单（合规条款响应点）
  - 图表需求清单（数量、类型、说明）
  - 证据要求（需要哪些背景材料支撑）
  - 依赖关系（哪些章节必须先写）

**关键能力**：
- 章节粒度自适应（粗细可调）
- 优先级排序：合规优先 → 技术 → 商务 → 加分项
- 依赖分析：识别章节间的前后约束
- 必含要素对齐：响应矩阵条款不能漏

## 2. 章节撰写器（Chapter Writer）

**输入**：
- 章节规格说明书
- 章节素材（从背景材料检索）
- 章节级硬约束（global_facts、术语表）

**输出**：
- 章节正文 Markdown
- 章节内嵌图表占位符（位置 + 类型 + 说明）
- 章节引用证据清单

**关键能力**：
- 章节级 Prompt 模板（不同类型章节用不同模板）
- 章节级证据约束（"没有证据不能写"）
- 章节级最小字数保证（每小节 ≥ 800 字）
- 章节级术语一致性（从知识库抽取术语表，强制使用）
- 章节内图表占位生成（图表描述 + 位置锚点）

## 3. 章节图表生成器（Chapter Illustrator）

**输入**：
- 章节正文（含图表占位符）
- 章节素材
- 章节规格里的图表需求清单

**输出**：
- 图表本体：Mermaid 源码 / 表格 HTML / AI 图 / 数据图表 JSON
- 图表元数据：编号、标题、说明、数据源、章节引用关系
- 渲染产物：PNG / SVG 文件

**关键能力**：
- 图表类型分类与适配策略（详见第四节）
- 图表与正文同步生成（不是后期插入）
- 图表修复链路（语法错误、风格不一致、数据矛盾）
- 图表缓存与重用（同章节多轮扩写时复用）

## 4. 章节一致性审计器（Chapter Consistency Auditor）

**触发时机**：
- 章节内：每章节撰写完成后立即审计（不阻塞其他章节）
- 跨章节：所有章节完成后统一审计

**审计维度**：
- 事实一致性：是否虚构资质、案例
- 术语一致性：同一概念在不同章节用同一术语
- 数据一致性：技术参数、商务承诺不矛盾
- 合规一致性：响应矩阵条款 ↔ 章节内容
- 图表一致性：图表数据 ↔ 正文引用

**输出**：
- 审计报告（按章节组织的问题列表）
- 自动修复建议（部分可自动修复）

## 5. 标书汇总器（Bid Assembler）

**输入**：
- 所有章节正文
- 所有图表（带渲染产物）
- 响应矩阵
- 一致性审计报告
- 用户的 Word 模板（可选）

**输出**：
- 完整 Word 标书（含目录、页眉、页脚、章节、图表、附录）
- 完整 PDF 标书
- 标书摘要页

**关键能力**：
- 模板套用（用户上传的 Word 模板 → 自动填充）
- 章节排序与编号
- 图表编号统一（图 1-1、图 1-2……表 1-1……）
- 交叉引用自动维护（"如图 3-2 所示"）
- 目录自动生成

# 四、图表体系设计（重点章节）

## 4.1 图表分类

### 按用途分类

| 类型 | 用途 | 典型例子 |
|---|---|---|
| 流程图 | 业务流程、审批流、工作流 | 投标流程、项目实施流程 |
| 架构图 | 系统架构、技术架构、部署架构 | 系统拓扑图、技术路线图 |
| 时序图 | 交互逻辑、调用关系 | 系统集成时序图 |
| 响应矩阵 | 合规条款 ↔ 响应内容 | 招标要求逐条响应表 |
| 数据图表 | 数据对比、趋势 | 柱状图、折线图、饼图 |
| 示意图 | 场景、界面、拓扑 | 业务场景图、组织架构 |
| 组织架构 | 人员、角色 | 项目组结构图 |

### 按生成方式分类

| 方式 | 适用类型 | 工具链 |
|---|---|---|
| Mermaid 源码渲染 | 流程图、架构图、时序图 | mermaid.ink + 本地 fallback |
| 结构化表格生成 | 响应矩阵、对照表 | HTML / JSON → docx 表格 |
| AI 图生成 | 示意图、封面图 | 多 provider fallback |
| 数据图表生成 | 数据图表 | 基于真实数据生成 |

## 4.2 图表元数据规范

每张图表必须有以下元数据：

```
{
  "id": "fig-3-2",
  "type": "flowchart | architecture | sequence | matrix | data | illustration | org",
  "chapter_id": "ch-3",
  "position": "inline | appendix",
  "title": "系统架构图",
  "caption": "本系统采用分层架构...",
  "source": "mermaid | ai-generated | data | table",
  "source_content": "flowchart TD\n  A-->B...",   // 源码 / prompt / HTML / JSON
  "data_refs": ["doc-company-profile", "tech-doc-01"],   // 数据来源引用
  "rendered_path": "figures/fig-3-2.png",                 // 渲染产物
  "version": 1,
  "status": "draft | approved | deprecated"
}
```

## 4.3 图表生成策略

### 策略 A：Mermaid 优先
- 适用：流程图、架构图、时序图
- 流程：LLM 生成 Mermaid 源码 → 校验语法 → mermaid.ink 渲染 → 失败降级本地 → 仍失败作为占位图
- 优点：可编辑、可缓存、可搜索、文件体积小
- 风险：弱模型经常输出语法错误

### 策略 B：响应矩阵走结构化
- 适用：响应矩阵、合规条款对照表
- 不走 Mermaid，直接生成 HTML / JSON 表格
- 强制从 RFP 抽取条款 ↔ 从章节正文抽取响应
- 一致性审计要校验"每条款都有响应、每响应都对应条款"

### 策略 C：AI 图走多 provider
- 适用：示意图、封面图、组织架构图
- 多 provider fallback（避免单 provider 失败）
- 风格约束（同一份标书的图风格要统一）
- 缓存（同类型图不重复生成）

### 策略 D：数据图表禁编造
- 适用：所有数据图表
- 必须有真实数据来源
- 数据来源进入证据链
- 无数据则不生成（绝不编造）

## 4.4 图表与正文的一致性

### 图表 → 正文引用
- 每张图表被正文引用至少一次（"如图 3-2 所示"）
- 引用关系在汇总阶段自动校验
- 未被引用的图表：警告（可能是冗余图）

### 正文 → 图表数据
- 正文描述的数据必须跟图表数据一致
- 一致性审计要校验数值、单位、时间
- 数据矛盾时标记为"需人工确认"

### 图表 → 证据链
- 每张图表的数据源进入证据链
- 证据链可追溯到具体背景材料
- 标书查重时图表也算（防止图也是抄的）

## 4.5 图表生命周期

```
draft → reviewing → approved → embedded → deprecated
   ↑         ↓
   └──── failed (重做)
```

- draft：刚生成
- reviewing：跟章节正文一起被审
- approved：审计通过
- embedded：已嵌入 Word
- deprecated：被新版本取代

# 五、核心数据结构

## 5.1 章节规范（chapter_specs）

```
ChapterSpec {
  id: string,                          // "ch-3"
  title: string,                       // "技术方案"
  level: 1 | 2 | 3,                    // 章节层级
  order_index: number,                 // 同级排序
  parent_id: string | null,            // 父章节
  chapter_type: "technical" | "business" | "compliance" | "price" | "case" | "management",
  target_word_count: number,           // 目标字数
  min_word_count: number,              // 最小字数
  writing_style: "narrative" | "list" | "comparison" | "mixed",
  required_elements: string[],         // 必含要素
  illustration_requirements: IllustrationRequirement[],  // 图表需求
  evidence_requirements: EvidenceRequirement[],          // 证据需求
  dependencies: string[],              // 依赖的章节 ID
  status: "planned" | "writing" | "illustrating" | "chapter_auditing" | "done" | "failed"
}
```

## 5.2 章节正文（chapter_contents）

```
ChapterContent {
  id: string,
  spec_id: string,
  version: number,                     // 版本号
  content_markdown: string,            // 含图表占位符
  embedded_illustrations: IllustrationRef[],
  citation_evidence: string[],         // 证据引用 ID
  word_count: number,
  min_word_met: boolean,
  generated_at: timestamp,
  generated_by: "ai" | "human" | "hybrid"
}
```

## 5.3 图表库（illustrations）

```
Illustration {
  id: string,
  chapter_id: string,
  type: "flowchart" | "architecture" | "sequence" | "matrix" | "data" | "illustration" | "org",
  position: "inline" | "appendix",
  order_in_chapter: number,            // 章节内顺序
  title: string,
  caption: string,
  source: "mermaid" | "ai-generated" | "data" | "table",
  source_content: string,              // Mermaid 源码 / AI prompt / 表格 HTML / 数据 JSON
  rendered_path: string | null,        // 渲染产物路径
  data_refs: string[],                 // 数据来源引用
  version: number,
  status: "draft" | "reviewing" | "approved" | "embedded" | "deprecated"
}
```

## 5.4 响应矩阵（response_matrix）

```
ResponseItem {
  id: string,
  requirement_id: string,              // 招标要求 ID
  requirement_text: string,            // 招标原文
  response_chapter_id: string,         // 响应该条款的章节
  response_summary: string,            // 响应内容摘要
  compliance_status: "compliant" | "partial" | "non-compliant" | "not-applicable",
  evidence_refs: string[],             // 支撑证据
  auditor_notes: string | null         // 审计备注
}
```

## 5.5 证据链（evidence_chains）

```
Evidence {
  id: string,
  source_type: "company-profile" | "historical-bid" | "tech-doc" | "case" | "personnel" | "qualification",
  source_ref: string,                  // 来源文件/记录
  content: string,                     // 证据内容
  used_in_chapters: string[],          // 被哪些章节引用
  used_in_illustrations: string[],     // 被哪些图表引用
  reliability_score: number            // 0-1 可靠度
}
```

## 5.6 存储建议

- 大字段（正文、图表源码）落文件系统
- 数据库存元数据 + 引用关系
- 文件命名：`chapters/{chapter_id}/content_v{n}.md`、`illustrations/{illustration_id}/source.{ext}`
- 版本化保留（每次重写留一份，便于对比和回滚）

# 六、端到端工作流

## 6.1 用户输入

### 必填
- 标书要求：RFP 文件（PDF/Word）+ 关键参数（项目名、采购方、截止时间）

### 选填
- 背景材料：
  - 公司资质（营业执照、ISO 认证、行业资质）
  - 历史标书（往期中标标书，3-5 份）
  - 技术文档（产品手册、技术白皮书）
  - 案例库（过往项目案例，含合同/验收单）
  - 人员信息（项目团队简历、证书）
- 模板：Word 模板（用户历史使用的格式）
- 偏好：写作风格、术语偏好、图表风格

## 6.2 自动处理流水线

```text
输入校验（文件格式、必填项）
   ↓
RFP 解析（结构化抽取：项目信息、章节要求、合规条款、评分规则）
   ↓
背景材料索引化（按章节类型建立索引）
   ↓
章节规划器（一次性，输出章节大纲 + 规格清单）
   ↓
   ┌────────────────────────┐
   │ 人在回路点 1：章节大纲确认 │
   └────────────────────────┘
   ↓
章节级并行处理（每章节独立流水线，可并发 10）
   ├─ 素材检索（按章节规格从背景材料取素材）
   ├─ 章节撰写（按章节规格生成正文 + 图表占位）
   ├─ 图表生成（按图表需求生成 Mermaid/表格/AI 图）
   ├─ 章节内一致性审计（事实/术语/数据）
   └─ 章节内最小字数校验
   ↓
跨章节一致性审计（响应矩阵交叉、术语统一、数据一致）
   ↓
废标项扫描（合规风险、错别字、逻辑矛盾）
   ↓
标书查重（与历史标书比对，避免雷同）
   ↓
   ┌────────────────────────┐
   │ 人在回路点 2：审计问题处理 │
   └────────────────────────┘
   ↓
标书汇总器（输出 Word/PDF，含目录/页眉/页脚/图表编号/交叉引用）
   ↓
   ┌────────────────────────┐
   │ 人在回路点 3：最终样式微调 │
   └────────────────────────┘
   ↓
完整标书输出
```

## 6.3 自动化等级映射

| 环节 | 自动化等级 | 说明 |
|---|---|---|
| RFP 解析 | L1 完全自动 | 三档解析策略（简单/中等/复杂） |
| 章节规划 | L1 + 人在回路确认 | 自动生成大纲，人确认后撰写 |
| 章节撰写 | L1 完全自动 | 每章节独立流水线 |
| 图表生成 | L1 + 多 provider fallback | Mermaid/表格/AI 图/数据图表 |
| 章节内审计 | L1 完全自动 | 事实/术语/数据 |
| 跨章节审计 | L2 人在回路 | 自动扫描 + 人工确认 |
| 废标项扫描 | L2 人在回路 | 自动扫描 + 人工处理风险点 |
| 标书汇总 | L1 + L3 微调 | 自动汇总，人微调样式 |

## 6.4 人在回路点详细说明

### 点 1：章节大纲确认
- **触发**：章节规划完成后、撰写开始前
- **用户动作**：浏览大纲、调整章节粒度、补充必含要素
- **不阻塞**：可跳过（用默认大纲）
- **典型耗时**：5-15 分钟

### 点 2：审计问题处理
- **触发**：跨章节审计 + 废标项扫描完成后
- **用户动作**：逐条处理审计问题（自动修复 / 人工修改 / 标记忽略）
- **必须**：用户主动确认才能进入汇总
- **典型耗时**：15-60 分钟

### 点 3：最终样式微调
- **触发**：标书汇总后
- **用户动作**：调整页眉页脚、字体、间距、图表位置
- **可选**：直接采用默认样式
- **典型耗时**：5-30 分钟

## 6.5 端到端性能预估

假设：50 章节、每章节 2000 字、每章节 3 张图

| 环节 | 串行耗时 | 并发耗时 |
|---|---|---|
| RFP 解析 | 30s | 30s |
| 章节规划 | 20s | 20s |
| 章节撰写 | 50 × 30s = 25min | 5-8min（10 并发） |
| 图表生成 | 50 × 15s = 12.5min | 3-5min（10 并发） |
| 跨章节审计 | 60s | 60s |
| 标书汇总 | 30s | 30s |
| **总计** | **~40min** | **~10min** |

加上人在回路点，端到端实际耗时约 **40-90 分钟**（取决于人工确认速度）。

# 七、关键技术难点

## 7.1 章节拆分的粒度

| 太粗 | 太细 | 经验值 |
|---|---|---|
| 单章节 > 5000 字 | 章节 < 500 字 | 800-3000 字/章节 |
| 撰写质量下降 | 跨章节一致性难 | 3 级标题 |
| 图表插入乱 | 章节间上下文断裂 | 每章节 2000 字左右 |

**自适应策略**：根据 RFP 总字数估算章节数 = ceil(总字数 / 2000)。

## 7.2 图表生成质量

| 图表类型 | 常见问题 | 缓解 |
|---|---|---|
| Mermaid | 语法错误率高（弱模型常见） | Mermaid 校验 + 修复链路（局部修补，不重发） |
| AI 图 | 风格不一致（多 provider 风格不同） | 固定 prompt 模板 + 风格约束 |
| 数据图表 | 编造数据 | 走"证据驱动"，无数据则不生成 |
| 响应矩阵 | 漏条款 / 漏响应 | 强制从 RFP 抽取 + 章节正文反向核对 |

## 7.3 跨章节一致性

并行撰写导致的事实漂移（不同章节对同一项目用了不同描述）。

**解决方案（三层防护）**：
1. **章节级硬约束**：global_facts 注入 Prompt（项目名、采购方、关键参数）
2. **跨章节审计**：术语表 / 数据表 / 响应矩阵交叉
3. **统一术语库**：从背景材料抽取，强制使用

## 7.4 端到端延迟

- 50 章节串行撰写 25 分钟（用户体验差）
- 章节级并行 + 段级串行 → 5-8 分钟（10 并发）
- 图表生成并发受限于图片生成 provider（通常 2-5 并发）
- 进一步优化：预热（先发 1-2 个章节的请求） + 渐进式展示（边生成边展示）

## 7.5 长文档 Prompt 缓存

章节撰写时，章节规格 + 章节素材 + global_facts 都是共享上下文。

- **必须**把这些放 Prompt 前部（缓存命中率高）
- 同章节多次扩写时，章节正文也要放前面（缓存前缀复用）
- 经验：缓存命中可降本 5-10 倍

## 7.6 弱模型 JSON 不稳定

- 章节规格、图表规格都是结构化数据
- 弱模型经常输出残缺 JSON
- 必须配 JSON 修复链路（不重发完整大文本）
- 修复策略：截断检测、缺失字段补全、尾部分隔符修复

## 7.7 章节重做的状态一致性

- 用户可能要求重做某一章节
- 重做时必须保留：证据链、跨章节引用、图表编号
- 重做不能影响其他章节的状态

# 八、产品形态

## 8.1 用户视角

| 步骤 | 动作 | 时长 |
|---|---|---|
| 1 | 上传 RFP + 背景材料 | 5-10 分钟 |
| 2 | 点击"开始生成"（可调：粗/中/细粒度） | 1 分钟 |
| 3 | 等待（进度可视化，每章节状态可见） | 10-30 分钟 |
| 4 | 审计确认（人在回路） | 15-60 分钟 |
| 5 | 得到完整标书 | 即时 |

## 8.2 进度可视化

- 章节进度条（planned / writing / illustrating / chapter_auditing / done）
- 图表进度条（draft / reviewing / approved / embedded）
- 审计问题列表（实时更新）
- 预估剩余时间

## 8.3 中间可干预

- 暂停整个流程（保留所有状态）
- 重做单个章节（从 planned 重新开始）
- 重做单个图表
- 人工编辑章节（编辑后跳过该章节的撰写）

## 8.4 MVP 路线（建议）

### 第一阶段（1-2 周）：最小可行
- 章节规划器（简单版：固定章节模板）
- 单章节撰写器（不分并发）
- 简单图表（Mermaid 流程图）
- Word 输出

### 第二阶段（2-4 周）：章节化
- 章节级并行（10 并发）
- 多类型图表（流程/架构/响应矩阵）
- 章节素材库（手动管理目录）
- 跨章节一致性（术语表）

### 第三阶段（4-6 周）：端到端
- 章节级一致性审计
- 跨章节审计
- 人在回路点
- 标书模板套用
- 端到端一键生成

### 第四阶段（持续）：行业化
- 多行业模板（IT/政府采购/工程）
- 多 provider fallback
- 企业知识库
- 团队协作

## 8.5 不建议一开始做

- ❌ 全自动无人参与（必须保留人在回路，合规要求）
- ❌ 多行业并行支持（先深后宽）
- ❌ 复杂 UI 系统（MVP 阶段进度可视化即可）
- ❌ 实时协作（单机单用户先打通）

# 九、工程实现参考

## 9.1 章节级任务组锁

- 每章节一个 group-exclusive 锁（防止同一章节被并发撰写）
- 章节间并行（不同章节不互锁）
- 跨章节审计用独立的 group（不阻塞章节并行）
- 人工编辑用 advisory lock（不阻塞其他流程）

## 9.2 章节状态机

```
planned → writing → illustrating → chapter_auditing → done
   ↑         ↓            ↓                ↓
   │       paused      paused           failed
   │         ↓            ↓                ↓
   │     restoring   restoring       writing (重做)
   │         ↓            ↓
   └───── planned (可恢复起点)
```

## 9.3 章节并行调度

- 章节队列：维护"待撰写 / 撰写中 / 已完成"三个集合
- 并发控制：可配置（默认 10）
- 优先级：合规章节 > 技术章节 > 商务章节 > 加分项章节
- 失败处理：单章节失败不阻塞其他章节，可重试

## 9.4 图表存储

| 类型 | 源数据 | 渲染产物 |
|---|---|---|
| Mermaid | 源码入库 | PNG/SVG 落文件 |
| 表格 | HTML/JSON 入库 | 渲染产物落文件 |
| AI 图 | prompt 入库 | 图片文件落文件 |
| 数据图表 | 数据 JSON 入库 | 渲染产物落文件 |

## 9.5 章节级 Prompt 模板

每章节类型一个模板（技术/商务/合规/价格/案例/管理），分三段：

```
[段 1] 系统级共享（global_facts、术语表、写作风格）  ← 所有章节共享
[段 2] 章节级规格（章节规格说明书）                  ← 本章节唯一
[段 3] 章节级素材（检索到的证据、上一章节摘要）       ← 本章节唯一
```

- 段 1 + 段 2 的前缀缓存（多章节共享）
- 缓存命中可降本 5-10 倍

## 9.6 跨章节审计实现

- 术语表抽取（从所有章节正文 + 背景材料）
- 响应矩阵交叉（每条款 ↔ 每章节）
- 数据一致性（技术参数、商务承诺的关键数值）
- 报告组织：按章节 + 按问题类型

## 9.7 标书汇总实现

```
Markdown → AST → docx
章节排序与编号（按 chapter_specs.order_index）
图表编号统一（图 1-1, 图 1-2... 表 1-1...）
交叉引用解析（"如图 3-2 所示" → 真实图表）
目录自动生成（基于章节大纲）
页眉页脚模板套用
```

## 9.8 端到端可观测性

- 每个环节耗时统计
- 每章节成功率、失败率、重试率
- 每章节 token 用量（输入/输出/缓存命中）
- 跨章节审计问题分布
- 人在回路点停留时长

# 十、风险与边界

## 10.1 端到端自动的边界

- 完全自动生成标书在法律/合规上仍有风险
- 必须保留"人在回路"出口（审计问题、最终样式）
- **标书最终签章前必须人工审核**（这是合规要求，不是技术问题）

## 10.2 图表生成成本

- AI 图每次调用 0.01-0.1 USD
- 50 章节 × 3 张图 = 150 次调用 = 1.5-15 USD/份标书
- 必须缓存（同一份标书内相似图不重复生成）
- 必须可降级（AI 图失败时降级为 Mermaid）

## 10.3 章节并行引发的状态一致性

- 必须用任务组锁（章节级 + 跨章节审计级）
- 章节重做时必须保留证据链（不能重做时丢失已用证据）
- 跨章节审计必须能容忍单章节重做

## 10.4 弱模型 JSON 不稳定

- 章节规格、图表规格都是结构化数据
- 必须配 JSON 修复链路
- 章节正文尽量用 Markdown（不是 JSON），降低修复负担

## 10.5 长 Prompt 成本

- 每章节撰写 Prompt 几万字（章节规格 + 素材 + global_facts）
- 必须用 Prompt 缓存（共享前缀）
- 必须用便宜模型做初稿、好模型做终稿（双层模型策略）

## 10.6 用户输入质量

- RFP 不规范（OCR 扫描件、加密 PDF）→ 解析失败
- 背景材料不全（缺资质、缺案例）→ 撰写时只能编造
- 缓解：
  - 输入校验（明显问题提前拦截）
  - 缺失背景材料时主动询问用户（而不是默默编造）
  - 解析三档策略（本地/中等/付费 API）

## 10.7 图表事实性风险

- AI 图可能生成不合规的视觉元素（公司 logo、人物肖像）
- 数据图表必须有审计痕迹
- 缓解：AI 图走"风格化插画"路线（避免具体元素），数据图表必须标注数据来源

# 十一、下一步建议

按优先级排序：

### 1. 章节规划器原型（1-2 周）
- 输入 RFP，输出章节大纲 + 规格清单
- 不做撰写，只验证"拆分粒度"是否合理

### 2. 单章节端到端验证（2-3 周）
- 选 1 个典型章节（比如"技术方案"）
- 走通：规格 → 撰写 → 图表 → 审计 → Word
- 验证：图表与正文一致性、章节字数、Prompt 缓存命中率

### 3. 章节级并发验证（3-4 周）
- 多章节并行（10 并发）
- 验证：跨章节一致性、任务组锁、性能

### 4. 完整端到端 MVP（4-6 周）
- 章节规划 → 并行撰写 → 跨章节审计 → 人在回路 → 汇总输出
- 验证：从 RFP 到完整标书的全流程

### 5. 行业化扩展（持续）
- 多行业模板
- 多 provider fallback
- 企业知识库
- 团队协作
